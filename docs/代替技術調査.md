# Markmiru 代替技術調査

最終更新: 2026-06-01

採用ライブラリ（[`利用ライブラリ一覧.md`](利用ライブラリ一覧.md)）について、代替候補を調査し、本アプリの要件（**軽量・高速起動・綺麗な閲覧・GFM＋mermaid・閲覧重視**／Wails WebView）に照らして比較する。

## 0. 結論サマリ

| 領域 | 採用 | 主な代替 | 判定 |
|------|------|----------|------|
| UI フレームワーク | Svelte | React / Vue / SolidJS | **Svelte 維持**（軽量・高速が MUST に合致） |
| JS ランタイム/PM | Node.js + npm | Bun / Deno / pnpm | **Node 既定。Bun は任意の DX 向上**（Deno は不適） |
| Markdown 解析 | markdown-it | marked / remark(unified) | **markdown-it 維持**（拡張性と速度の均衡） |
| 構文ハイライト | highlight.js | Shiki / Prism | **highlight.js 既定。Shiki は品質重視の選択肢** |
| ソースエディタ | CodeMirror 6 | Monaco / Ace | **CodeMirror 6 維持**（軽量。Monaco は重すぎ） |
| HTML サニタイズ | DOMPurify | sanitize-html / xss | **DOMPurify 維持**（ブラウザ標準） |
| 図表 | mermaid | （実質代替なし） | **mermaid 必須**（要件が mermaid 指定） |
| ビルドツール | Vite | webpack / Parcel / esbuild | **Vite 維持**（Wails 既定・高速） |
| アプリ基盤 | Wails | Tauri / Electron | 決定済み（[`技術選定.md`](技術選定.md)） |

## 1. UI フレームワーク：Svelte vs React（重点）

### 比較

| 観点 | Svelte（採用） | React |
|------|----------------|-------|
| バンドルサイズ | **2〜5KB**（gzip）。実アプリでも React 比 約3.8倍小 | 約42KB（gzip） |
| 実行速度 | React 19（Compiler なし）比 **約40%高速**、Compiler ありでも約20%高速 | Compiler で改善するが及ばず |
| メモリ | 1,000行で約4.2MB（**約半分**） | 約8.8MB（自動メモ化込み） |
| 起動(TTI) | **30〜40%高速** | 遅い |
| 学習コスト | 低い（素の Web に近い・記述量少） | フック/ライフサイクル等で高め |
| エコシステム/採用 | 小さめ | **圧倒的に大**（350k+ パッケージ・求人多） |

### 本アプリでの評価

- 本アプリの中核ライブラリ（**markdown-it / mermaid / highlight.js / CodeMirror 6 はいずれもフレームワーク非依存**）であり、React の最大の強み「巨大なエコシステム/React 専用コンポーネント」が**ほとんど効かない**。
- 一方、本件は「**軽量・高速起動（MUST）**」が要。Svelte の小バンドル・高速・低メモリは WebView アプリに直接効く。
- **結論: Svelte 維持が最適。** React を選ぶ合理性は「チームが React に深く習熟している」場合に限られる。
- 補足: **SolidJS**（JSX 記法＋Svelte 並みの軽量・高速）、**Vue 3**（中間）も候補。Svelte が合わなければ SolidJS が次点。詳細は §1.1。

### 1.1 深掘り：Svelte 5 vs SolidJS

両者は **コンパイラ志向＋シグナルベースの細粒度リアクティビティ**という同系統。違いは主に**記法（テンプレート vs JSX）・エコシステム規模・成熟度**で、**性能・サイズは本アプリ規模では体感差なし**。

| 観点 | Svelte 5（採用） | SolidJS |
|------|------------------|---------|
| ランタイム(gzip) | **約1.6KB** | 約7KB |
| 生の実行性能 | 非常に高速 | **わずかに上**（差は95%のアプリで体感不可） |
| リアクティビティ | runes ＋ テンプレート | signals ＋ JSX |
| 週間DL / Star | **約2.7M / 86K**（約4倍大） | 約1.49M / 35K（YoY +60%） |
| 成熟度 | **5 は 2024/10 安定**、保持率91%（最高） | 1.x 安定。**2.0 は 2026/5 時点ベータ** |
| UI 部品 | 豊富 | Kobalte / Ark UI / Solid UI 等、増加中だが少なめ |

**Svelte 5 — メリット**
- エコシステムが約4倍大きく、チュートリアル・UI 部品・解決事例が豊富 → 詰まりにくい。
- 学習コストが低い（テンプレート記法）。State of JS で DX 1位・保持率91%。
- バッテリー同梱（スコープ CSS・トランジション・store）。
- ランタイム最小（約1.6KB）で WebView アプリに最適。
- 本プロジェクトの設計・文書がすでに Svelte 前提。

**Svelte 5 — デメリット**
- runes 導入時に賛否の議論。
- コンパイラの「魔法」的挙動が一部直感に反する場合。
- 一部ニッチ用途で Svelte 5 対応ライブラリの成熟が途上との報告。

**SolidJS — メリット**
- 生の実行性能は最速クラス（細粒度シグナルが最効率）。
- JSX 採用で React 経験者が移行しやすい。
- リアクティビティが非常に明示的で、プレーン JS に近い実行時意味論（予測しやすい）。
- ランタイムも十分小さい（約7KB）。

**SolidJS — デメリット**
- エコシステム・コミュニティが小さい（資料/事例が少なく、エッジケースで自力解決が増える）。
- 2.0 がベータ進行中（first-class async）で、安定運用は 1.x。今着手だと 2.0 移行の見極めが必要。
- JSX のためスコープ CSS・トランジション等は別途対応が必要。

**本アプリでの結論**
- 性能・サイズは両者とも要件を余裕で満たし決め手にならない（中核ライブラリはフレームワーク非依存）。
- 決め手はエコシステム・DX・成熟度で、**Svelte が優位**。
- **Svelte 維持を推奨**。SolidJS は「JSX 嗜好」かつ「2.0 ベータの流動性を許容」できる場合の対抗馬。

## 2. JS ランタイム/パッケージマネージャ：Node.js vs Deno vs Bun（重点）

### 最重要の前提

**この選択はビルド時のみに影響し、出荷アプリの起動速度・サイズには影響しない。** 出荷物は OS 標準 WebView 上で動くコンパイル済み Svelte であり、Node/Deno/Bun は同梱されない。さらに **Wails は `wails.json` の `frontend:install` / `frontend:build` コマンドを差し替えるだけ**で PM/ランタイムを変更でき、**バンドラは引き続き Vite** のままになる。つまり差は純粋に**開発体験（インストール/ビルド速度・互換性）**の問題。

### 比較

| 観点 | Node.js + npm（採用） | Bun | Deno |
|------|----------------------|-----|------|
| 成熟度/互換 | **最高**。全ツール・全プラグインが動く。Wails 例も npm 前提 | Node API **約99.7%互換**（Bun 2.0） | npm **約95〜98%互換**（Deno 2） |
| install 速度 | 遅い（中規模 約20秒） | **約1秒（10〜20倍速）** | 中（約17秒） |
| Vite との相性 | ◎ 完全 | ○ `bunx --bun vite` で利用可。2026 の定番は「Bun=PM/ランナー＋Vite=バンドラ」 | △ フロントのプラグイン群と摩擦が出やすい |
| Wails との相性 | ◎ 既定 | ○ `wails.json` と `package.json` の scripts を bun に変更すれば可（dev watcher は要確認） | △ フロントビルド用途では利点が薄く摩擦が多い |
| リスク | なし | 小（稀な Node 内部 API/一部プラグイン） | 中（フロント生態系との相性） |

### 本アプリでの評価

- **Node + npm を既定**とする（ゼロリスク・全ドキュメントが前提）。
- **Bun は任意の DX 向上**として有力。install/build が劇的に速く、Wails も設定変更で対応可。CI/開発を速くしたいなら採用価値あり（小さな互換リスクは許容範囲）。pnpm（高速・省ディスク）も同様に差し替え可能な選択肢。
- **Deno はフロントビルド用途には不向き**（Vite 生態系との摩擦が大きく、得られる利点が少ない）。
- 繰り返しになるが、**いずれを選んでも出荷アプリの「起動が素早い（MUST）」には影響しない**。

## 3. Markdown 解析：markdown-it vs 代替

| 候補 | 特徴 | 本件評価 |
|------|------|----------|
| **markdown-it（採用）** | プラグイン生態系が最大。速度と拡張性の均衡。GFM 対応、mermaid/タスクリスト統合が容易 | **維持**。拡張余地（将来の数式・脚注等）に有利 |
| marked | 最速・最小（~50KB）・依存なし。拡張は限定的 | 最小構成にしたい場合の代替 |
| remark / unified | AST ベースで最強の変換力（MDX 等）。重く複雑 | ビュワーには過剰。重い変換が要るなら |
| micromark | 低レベル・高準拠。remark の内部 | 直接利用は非推奨 |

---

## 4. 構文ハイライト：highlight.js vs Shiki vs Prism

| 候補 | 特徴 | 本件評価 |
|------|------|----------|
| **highlight.js（採用）** | ゼロ設定・言語自動判定・クライアント動作・シンプル（BSD） | **既定として妥当**。必要言語のみ同梱で軽量化 |
| Shiki | **VS Code 同等の高品質**（TextMate 文法・正確なテーマ）。ただしビルド時想定・**Prism比約7倍遅い**・非同期・重い | 「**綺麗に表示（MUST）**」を最重視するなら有力な格上げ候補。クライアント再描画ではデバウンス前提で重さに注意 |
| Prism | 最速・軽量・プラグイン式。言語は手動ロード | より軽量にしたい場合 |

> **論点**: Shiki は見栄えが最良で「綺麗にレンダリング」の方針に合うが、軽量・高速とトレードオフ。閲覧品質を優先するなら Shiki への切替を検討する価値がある。

## 5. ソースエディタ：CodeMirror 6 vs Monaco vs Ace

| 候補 | バンドル(最小 gzip) | 特徴 | 本件評価 |
|------|--------------------|------|----------|
| **CodeMirror 6（採用）** | 約124KB | 軽量・高速・モバイル可・高カスタマイズ・Markdown 言語対応 | **維持**。「メモ帳的な軽い編集」に最適 |
| Monaco | **2MB超** | VS Code エンジン。補完等が強力だが重く、モバイル不可 | 過剰。軽量・高速 MUST に反する |
| Ace | 約98KB | 旧来型。可もなく不可もなく、CM6 ほど現代的でない | 非推奨（CM6 が上位互換的） |

## 6. HTML サニタイズ：DOMPurify vs 代替

| 候補 | 特徴 | 本件評価 |
|------|------|----------|
| **DOMPurify（採用）** | ブラウザ DOM ベースで高速・堅牢。cure53 製の事実上標準 | **維持**。WebView（ブラウザ）文脈に最適 |
| sanitize-html | Node 寄り（htmlparser2）。設定豊富だがブラウザでは重め | サーバ向き |
| xss (js-xss) | 軽量・ホワイトリスト式 | DOM clobbering 等への堅牢性は DOMPurify が上 |

---

## 7. 図表：mermaid（実質代替なし）

要件が「**mermaid を利用した図表**」と指定しているため、mermaid 構文をレンダリングできる代替は実質存在しない（PlantUML 等はサーバ/Java 必要で不適）。**mermaid は必須**。

---

## 8. ビルドツール：Vite vs 代替

| 候補 | 特徴 | 本件評価 |
|------|------|----------|
| **Vite（採用）** | 高速 dev（esbuild 事前束＋ネイティブ ESM）、本番は Rollup/Rolldown。Svelte 統合良好・Wails 既定 | **維持**。Vite 8 は本番束を Rust 製 Rolldown へ移行中 |
| webpack | 高機能だが設定が重く遅い | 非推奨 |
| Parcel | ゼロ設定だが制御性低 | 小規模向け代替 |
| esbuild 単体 | 超高速だが機能少 | 単体では不足 |

## 10. 総括

- **UI=Svelte / エディタ=CodeMirror 6 / サニタイズ=DOMPurify / ビルド=Vite / 図表=mermaid** は、本アプリの要件に対し**現行採用が最適**で、変更の動機は乏しい。
- **再考の余地があるのは2点**:
  1. **構文ハイライトを Shiki に**（綺麗さ最優先なら。軽量性とトレードオフ）。
  2. **PM/ランナーを Bun（または pnpm）に**（開発・CI 速度の DX 向上。出荷物への影響なし・低リスク）。
- **Deno はフロントビルド用途には不適**。
